PROMPT – Ajuste Manual de Horário na Finalização da Atividade

Você é um desenvolvedor sênior responsável por evoluir o meu sistema de agendamento e execução de atividades técnicas, que já está rodando no Replit.

O sistema já possui:

Início de atividade

Conclusão de atividade

Pop-up de confirmação (“Trabalho realizado? SIM / NÃO”)

Registro automático de horário de início e término

Dashboard e lógica operacional já funcionando

Seu objetivo agora é adicionar uma função de correção manual de horário de início e término da atividade no momento da conclusão, pensando principalmente em técnicos mais velhos que podem esquecer de iniciar/concluir no tempo exato.

1. O que deve ser implementado
1.1. Novo comportamento na conclusão da atividade

Hoje, ao clicar Concluir, aparece a pergunta:

“Trabalho realizado? SIM ou NÃO”

Você deve evoluir esse comportamento da seguinte forma:

🔹 Se o técnico clicar em “NÃO”

Mantém o fluxo atual (atividade encerrada como não realizada, motivo etc.)

Nada muda neste caminho.

🔹 Se o técnico clicar em “SIM”

Imediatamente após isso, deve abrir um novo pop-up/modal com estes campos:

Horário de Início (preenchido automaticamente com o horário registrado pelo sistema)

Horário de Término (preenchido automaticamente com o horário atual)

Ambos os campos devem ser EDITÁVEIS.

1.2. Requisitos do modal de ajuste

O modal deve conter:

Título: "Ajustar horário da atividade"

Campos:

Hora de início – tipo datetime ou time (usar o mesmo componente padrão do sistema)

Hora de término – tipo datetime ou time

O sistema deve validar:

O horário de término não pode ser menor que o horário de início

Ambos devem ser horários válidos

Botões:

Salvar

Cancelar (volta para o estado anterior sem alterar horários)

1.3. Gravação dos dados

Ao salvar:

O backend deve receber:

ID da atividade

Horário de início ajustado

Horário de término ajustado

Técnico responsável

Eventuais metadados atuais da conclusão (status, tipo da atividade, etc.)

Ajustar o endpoint existente de conclusão para aceitar os dois horários manualmente.

Não criar endpoints duplicados.
Apenas expandir o endpoint atual para aceitar:

{
  "start_time_adjusted": "...",
  "end_time_adjusted": "..."
}

1.4. Registro e auditoria

Ao salvar:

O horário original deve ser registrado internamente como “sugestão do sistema”

O horário ajustado deve ser registrado como “valor final informado pelo técnico”

Se houver tabela de auditoria, gravar:

Antes

Depois

Data/hora da alteração

ID do técnico

2. Ajustes de Interface (frontend)
2.1. Modal responsivo e simples

O técnico é uma pessoa mais velha → usar campos grandes, leitura fácil:

Inputs grandes

Botões largos

Visual limpo

Fonte padrão do sistema

2.2. Fluxo visual sugerido

Técnico clica Concluir

Pop-up padrão: "Trabalho realizado?"

Se SIM →

Abrir o modal de ajuste de horários

Ao salvar →

Fechar modal e concluir atividade

3. Requisitos obrigatórios

Manter a stack atual (não trocar libs, frameworks ou arquitetura)

Reutilizar:

Design

Componentes existentes

Endpoints e serviços já implementados no fluxo de conclusão

Não quebrar o fluxo atual de:

Dashboard

Relatórios

Indicadores

Não alterar a lógica de contagem de deslocamento ou execução já existente
(apenas ajustar o registro dos horários)

4. Entregáveis Esperados

Modal completo para edição de horários

Ajuste no fluxo de conclusão

Ajuste no endpoint de encerramento

Validação de horários

Registro auditável da alteração

Atualização automatizada do dashboard e relatórios com base no horário ajustado

5. Instruções finais

Implemente isso passo a passo dentro do meu projeto atual, mantendo:

Código limpo

Lógica clara

Boa usabilidade para técnicos mais velhos

Segurança e consistência na gravação dos horários

Essa funcionalidade deve ser integrada SEM quebrar as funcionalidades já existentes.
