========================
ETAPA 1 — INICIAR NAVEGAÇÃO + REGISTRAR TEMPO DE IDA
========================
Requisito:
- Ao clicar em “Iniciar navegação”, o sistema abre o GPS/navegador e captura (se possível) uma estimativa de tempo (ETA) para o destino da atividade.
- Antes do técnico iniciar a atividade (ou conforme UX atual, imediatamente após abrir a navegação), deve aparecer o modal de registro do tempo de deslocamento IDA.
- No modal IDA:
  - Exibir em destaque “Estimativa do GPS: X min” (quando houver).
  - Campo “Tempo realizado (min)” pré-preenchido com X, mas EDITÁVEL.
  - Manter opções atuais de “tipo de deslocamento”.
  - Ao confirmar, registrar deslocamento_ida para a atividade.
- Também definir no plano quais timestamps devem ser gravados:
  - navigation_started_at (clique iniciar navegação)
  - e como isso se relaciona com execution_started_at (início real da atividade)

O que quero no plano:
- Quando exatamente o modal abre (antes de “Iniciar atividade”, ou “Iniciar navegação” já inicia um estado “navegando”?)
- Como guardar o ETA do GPS (campo dedicado vs metadata).
- O que acontece se não houver ETA (sem chave / sem sinal / erro).

========================
Requisito:

- O modal deve trazer “Tempo estimado de execução” calculado automaticamente pelo sistema usando:
  - (agora - execution_started_at)
- Pré-preencher o campo “Tempo realizado (min)” com essa estimativa e permitir edição.
- Ao confirmar, registrar o tempo de execução e mudar status para concluída.

O que quero no plano:

- Como garantir que execution_started_at é sempre definido (e o que fazer se estiver nulo por alguma exceção).

========================
ETAPA 3 — PÓS-CONCLUSÃO: VOLTA (2 possibilidades) SEM IMPACTAR PRÓXIMA IDA
========================
Contexto:
Após concluir a atividade, existem duas realidades:
A) O técnico volta para a BASE
B) O técnico segue para a PRÓXIMA atividade da agenda

Regra central:
- Se ele seguir para a próxima atividade, NÃO registrar “volta” e NÃO reaproveitar esse deslocamento como IDA automática da próxima.
- O deslocamento IDA de uma atividade só pode ser registrado quando o técnico clicar “Iniciar navegação” daquela atividade.

Requisito para VOLTA PARA BASE:
- Se escolher voltar para base, abrir GPS para a base e capturar ETA (se possível).
- Exibir modal “Tempo de retorno à base” semelhante ao de IDA:
  - mostrar ETA em destaque
  - tempo editável
  - tipo de deslocamento
- Registrar esse tempo como “retorno_base”, vinculado à atividade que acabou de ser concluída, e também relacionado à base do técnico.

O que quero no plano:
- Propor um fluxo eficiente e simples para essa decisão pós-conclusão.
Sugestão: um modal “Próximo passo” com 3 botões:
  1) Voltar para base (gera retorno_base)
  2) Ir para próxima atividade (não gera retorno)
  3) Encerrar/Disponível (não gera retorno)
Mas você pode sugerir alternativa melhor, desde que preserve a regra central e fique fácil para o técnico.

MODELAGEM DE DADOS (PROPOSTA NO PLANO)
========================
Apresente 2 opções e recomende uma:

OPÇÃO 1 — Tabela de registros de tempo tipada (forte e simples):
- Uma tabela “activity_time_records” (ou similar) com:
  - activity_id
  - type ENUM: 'ida', 'execucao', 'retorno_base'
  - minutes_reported
  - gps_eta_minutes (nullable)
  - deslocamento_tipo (nullable)
  - started_at/finished_at (se fizer sentido)
  - created_at
- Regras: retorno_base sempre exige base_id; ida e execucao não.

OPÇÃO 2 — Event Sourcing leve (mais flexível):
- Uma tabela “activity_events” com:
  - activity_id
  - event_type ('NAV_START','TRAVEL_IDA_RECORDED','EXEC_START','EXEC_FINISH','RETURN_BASE_RECORDED','NEXT_ACTIVITY_SELECTED', etc.)
  - payload JSON (para guardar gps_eta, minutos, tipo deslocamento, base_id)
  - created_at
- Benefício: impossível misturar conceitos, e ótimo para auditoria e BI.

No plano, explique trade-offs e qual escolher para ASTEC agora, mantendo compatibilidade com o que já existe.
